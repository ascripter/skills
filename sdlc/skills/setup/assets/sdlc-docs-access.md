# SDLC Document Access — slice, don't slurp

The SDLC specs under `docs/` (notably `docs/PRD.yaml`, `docs/DATA-MODEL.yaml`
and `docs/TASKS.json`) grow large enough that loading one whole burns a big
slice of an agent's context
window. They are consumed by agents with limited context, so retrieve only the
structure plus the minimum relevant slice. `Read` with `offset`/`limit` loads
**only** the requested lines — the file's total size is irrelevant once you know
the range.

## The entrypoint: `docs/INDEX.yaml`

A **generated** location map **plus a cross-reference graph** — never hand-edit
it. It is regenerated by the `Write|Edit|MultiEdit` PostToolUse hook on every
canonical `docs/*.yaml` or `docs/TASKS*.json` edit — shard edits refresh the
`shards:` inventory — (or by running `python .claude/sdlc/docs_index.py`). It
contains these blocks:

- `generated_from` — per-file `sha256` prefix + line count (staleness guard).
- `sections` — for each canonical doc (YAML **and** the JSON canonicals
  `TASKS.json` / `CODE-MANIFEST.json`), every top-level key → `[start, end]`
  line range.
- `shards` — every `docs/<PARENT>__<slug>.yaml|json` sub-artifact present,
  keyed by its parent canonical (`UX.yaml: [UX__login.yaml, …]`). Use it to
  *discover* which per-surface/container/resource files exist — then load the
  shard whole (most are small by design). **Exception:** a large
  `TASKS__<cid>.json` (a container with many tasks) is NOT small — do not load it
  whole; its tasks are per-task addressable in `symbols` (below), so slice one
  task by id.
- `symbols` — **grouped by file**; each addressable symbol is one positional row:
  `name: [start, end, kind, context, summary]` (`context` is `~` when none). The
  owning file is the group header the row sits under — read the first two numbers
  as the `[start, end]` line range. Symbols cover DATA-MODEL `entities` + `enums`,
  PRD `FR-###` items and the other single-namespace id families
  (`NFR/WKF/INT/AIF/OOS/PER/…`), each `conventions.*` sub-block (kind
  `convention`) + `conventions.stage_dossiers` entries, UX `SCR-###` surfaces,
  canonical `TASKS.json` `TSK-###` tasks (bare id), and every
  `TASKS__<cid>.json` shard's tasks keyed by **qualified id** `<cid>/TSK-NNN`
  (matching how sdlc-code addresses tasks — the canonical file and every shard
  restart their numbering, so the `<cid>/` prefix disambiguates).
- `referenced_by` — the **inbound edge map** (edit blast-radius): for each defined
  corpus id, the symbols / sections / tasks that mention it, grouped by the doc
  that DEFINES the id. Consult this before editing a symbol so you reconcile every
  inbound site in the same pass.
- `dangling` — references whose id is a corpus family (`FR/NFR/ENT/WKF/SCR/…`) but
  resolves to no definition (a typo or a deleted symbol). `dangling: []` is the
  clean state; `docs_index.py --check` exits non-zero when it is non-empty.

`INDEX.yaml` duplicates no field bodies; `summary` is a ≤120-char hint only. The
canonical truth always lives in the source file — resolve there before relying
on it.

## Retrieval protocol

1. **Never read `PRD.yaml`, `DATA-MODEL.yaml`, or `TASKS.json` in full.** Load
   `docs/INDEX.yaml` first.
2. **Resolve by symbol, read by range.** Look the name up in `symbols`, then
   `Read(file, offset=start, limit=end-start+1)`. Use `summary` to pick the right
   symbol without opening several.
3. **Resolve a top-level block by its `sections` range.** To read one block of a
   doc (e.g. PRD `conventions`, DATA-MODEL `cross_references`), slice the
   `[start, end]` range listed under `sections.<file>.<key>` — don't load the file
   whole to find it.
4. **Follow cross-references lazily.** Only resolve a linked entity when the task
   needs its fields. To traverse relationships, slice the `composition` /
   `cross_references` sections (via their `sections` ranges) — do not load
   `entities` whole to find a link.
5. **Never inline large repeated blocks** into prompts or other artifacts; reference
   a symbol by name + range.
6. **Self-heal a stale range.** If a sliced range does not begin with the expected
   `<name>:` anchor, the index lagged an edit: relocate with `Grep`
   (`^\s*<name>:` for entities/enums/dossiers; `"<FR-id>:` for an FR), `Read`, then
   regenerate with `python .claude/sdlc/docs_index.py`.

## Power tools

`python .claude/sdlc/docs_index.py <cmd>`:

- `--show <symbol>` prints exactly one symbol's slice (`<symbol>` = an entity/enum
  name, an `FR-###`/`NFR-###`/… id, a `SCR-###` surface, a canonical `TSK-###` id,
  a shard task's qualified id `<cid>/TSK-NNN`, a `conventions.*` block key, or a
  `conventions.stage_dossiers` map key). Equivalent to looking the symbol up in
  `INDEX.yaml` and reading its range — the cheapest way to pull one task out of a
  large `TASKS__<cid>.json` shard.
- `--refs <symbol>` prints a symbol/id's **blast-radius**: the ids it references,
  the containers that reference it (`referenced_by`), and any dangling references
  inside it. **Run this before editing a symbol** so you reconcile every inbound
  site in the same pass.
- `--check` rebuilds the index and **exits non-zero if any reference is dangling**
  — the id-integrity gate for a pre-commit / CI hook. (The PostToolUse hook path
  stays non-blocking.)
- `--find kind=… context=… file=… text=… references=… referenced_by=…` is a
  predicate search over the symbol table (AND semantics) — e.g.
  `--find kind=surface` or `--find references=FR-001`.

## Maintaining the index

- `INDEX.yaml` is a derived artifact — edit the canonical source; the hook refreshes it.
- The generator (`.claude/sdlc/docs_index.py`) content-*sections* the canonical
  docs only. Per-surface / per-container sub-artifacts (`UX__*.yaml`,
  `ARCH__*.yaml`, `API__*.yaml`, `TASKS__*.json`), versioned snapshots, and
  `*_draft` files are not sectioned — most are independent documents small
  enough to load whole — but present shards ARE enumerated in the `shards:`
  block for discovery, and writing one refreshes the index. **The one content
  exception:** `TASKS__<cid>.json` shards are additionally per-task
  symbol-indexed (`<cid>/TSK-NNN` rows under `symbols:`) because a many-task
  container is too large to load whole.
- Do **not** split `PRD.yaml` / `DATA-MODEL.yaml` / `TASKS.json`: range-based
  slicing already gives
  free partial reads, and the `entities` graph (`composition`, `cross_references`)
  is a whole-file relation that splitting would fragment. (UX/ARCH/API/TASKS are
  sharded because their surfaces/containers/resources are independent documents.)
